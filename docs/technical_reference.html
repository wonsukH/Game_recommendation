게임 추천 에이전트 — 기술 레퍼런스

하이브리드 AI 에이전트: 행동 기반 CF(moat) + LLM 오케스트레이션 + 콘텐츠 스티어링. 모든 설계 결정이 사전등록 실험으로 검증됨(귀무·패배 포함 정직 보고).
⚠️ 갱신 메모 (2026-06): 데이터 층이 리뷰-CSV → 행동 SQLite(data_collection/steam.db, crawl_unified.py)로 재구축 중이다(로드맵 P4–P9: docs/ROADMAP.md). 본문의 카탈로그 수치(9,956 게임·447 태그·일부 제약 카운트)와 steam_appdetails.csv 메타 경로는 재구축 이전 스냅샷이라 바뀔 수 있다. 또한 개인화 CF는 LLM-with-library를 이기는 동시에 EASE(전통 recsys)와는 동률 — 즉 moat는 "개인화 한정·EASE 동률·데이터 의존"으로 범위 한정된 입증 우위다.

방향 전환 배경 — 왜 "개인화"인가

이 프로젝트는 처음엔 "자연어로 게임을 추천"하는 일반 추천기였다. 그러나 검증 과정에서 그 방향은 프론티어 LLM을 못 이긴다는 게 반복적으로 확인됐고, 데이터가 가리킨 한 방향 — 행동 기반 개인화 — 으로 목표를 재설정했다. 아래는 "현재 시스템의 한계"가 아니라 이 재설정을 이끈 발견들이다(폐기·위임한 방향).

버린/위임한 방향발견 (증거)그래서
익명 자연어 추천블라인드 평가에서 시스템이 LLM에 ~96% 패배(P2e/P2f), LLM 환각도 2.5%뿐경쟁하지 않고 anonymous는 LLM에 위임
"숨은 명작 발굴" 헤드라인시스템의 niche 픽은 "유명하지 않을 뿐 더 좋지도 않음"으로 판정(P2e/P2f)헤드라인 폐기(품질 게이팅+파워유저 전제 시에만 가능)
PPMI+SVD / W_align 임베딩tag-cosine보다 유의하게 나쁘거나(P1, p=1.5e-17) 고장(P2b) — 복잡도값 못 함드롭, 단순 tag-cosine 채택
크롤 풀을 moat로크롤된 카탈로그는 강점이 아니라 제약(LLM은 풀 밖도 앎)moat는 카탈로그가 아니라 행동 신호(CF)로 재정의
전환점(P3): 유저 자신의 라이브러리/행동에서 추천할 때는 행동 CF가 "LLM+라이브러리"를 유의하게 이긴다(recall@20 +0.120, sig). 이것이 커스텀 시스템이 정당성을 얻는 유일한 방향 → 현재 목표 = 개인화 추천 에이전트.

상세 여정은 전체 실험 결과(P0–P2f)와 DELIBERATION_LOG.md 참조. 이 섹션의 항목들은 의도된 설계 결정의 근거이며, 현재 시스템의 결함이 아니다.

설계 철학

① 측정 먼저

기능을 만들기 전에 항상 진단·지표 검증부터. 지표 자체가 믿을 만한지(floor/ceiling·교란·판별력) 먼저 평가한 뒤에만 사용.

② 사전등록 결정규칙

"무엇을 보면 채택/기각"을 실험 전에 고정. 결과가 애매해도 골대를 옮기지 않음.

③ 귀무·패배 보고

안 되는 것도 그대로 기록. D3(shrinkage)·D1(recall null)·익명추천 96% 패배 전부 명시.

④ 비순환 검증

평가 라벨이 추천기와 독립이어야 함. co-play 행동 라벨, 신장르 holdout 등 비순환 우선.

⑤ 적대적·블라인드 교차검증

독립 judge(Claude/Gemini)를 A/B 블라인드로. familiarity-bias 중립화 설계.

⑥ 모든 수치 CI

bootstrap·paired-bootstrap 신뢰구간. 점추정만으로 결론 내지 않음.

아키텍처 (LangGraph)

요청은 단일 StateGraph를 통과한다. 라우터가 유형을 정하고, 각 유형이 실험으로 검증된 엔진으로 분기한 뒤, 공통 후처리(제약 검증 → critic/refine 사이클 → 설명)로 합류한다.

router_node (LLM 분류 + 상태 guard) │ request_type 확정 ├─ library → library_node ─┐ CF 개인화 + 콜드폴백 ├─ seed → seed_node │ seed-CF (프랜차이즈 제외) ├─ multi_entity → multi_node ├─→ filter_node ──→ critic_node ├─ explore → steer_node │ (제약+played) │ 미달 & 예산↑ └─ anonymous → anonymous_node ─┘ │ → 제약완화 → 재필터 (사이클) └─→ response_node (LLM 설명) → END

구현: serving/agent_graph.py · langchain_core + langchain_google_genai(Gemini) + langgraph. 자원(CFRecommender, ContentLayer, HybridRecommender, CatalogMeta, LLM)은 빌드 시 1회 생성해 노드 클로저로 주입. 메모리(본 게임/라이브러리)는 Streamlit 세션 상태.

명시적 라우팅

라우팅은 2단계 + 디스패치다. ②③이 LLM 판단이 아니라 결정론적 코드라는 점이 "명시적"의 핵심.

① LLM 분류 — 6가지 request_type

LLM이 유형을 찍고 동시에 constraints(협동/멀티/한국어/무료/최대가격/출시연도)와 seed_titles(공식 영어 제목)도 추출한다.

multi_entity친구/타인 언급"나랑 친구 둘 다 할"
explore NEW취향에서 벗어나려는 의도(새 장르/색다른/특정 측면)"안 해본 장르로", "전투 좋았는데 다른 분위기"

② 상태 기반 guard (결정론적 오버라이드)

LLM을 그대로 믿지 않고 실제 가진 데이터로 코드가 덮어쓴다. "explore/library/multi는 라이브러리가 있어야 성립" 같은 전제를 코드가 보장.

explore  & 라이브러리 없음    → anonymous   (벗어날 *기준*이 없음)
anonymous & 라이브러리 있음   → library     (가능하면 항상 개인화)
파싱 실패                      → anonymous   (안전 폴백)

엔진별 상세

CF moat — 개인화 협업필터링 cf_recommender.py

검증된 핵심(moat). 1.2M 유저 공출현 행렬 기반 item-item CF, 플레이타임 가중.

  • "좋아요" = s_round10_rec ≥ 7 (추천 풀로 제한)
  • 유사도 = conditional cosine C[i,j]/√(deg[i]·deg[j]), support 바닥 C ≥ 3
  • 점수 score(g)=Σ w_p·sim(g,p), 취향 가중 w_p=log(1+playtime_p/평균플레이타임) (평균보다 오래 = 강한 취향 신호)

콘텐츠 레이어 — 태그 코사인 content.py

game×tag(9,956×447, 이진) 행렬을 L2 정규화한 코사인(검증된 Vb). 단일 게임이 아니라 플레이타임 가중 라이브러리 태그 프로파일로 일반화 → 전 풀(콜드 포함) 점수화. 임베딩 투영(W_align)이 아님 — 그건 실험에서 패배(P2b/F1).

하이브리드 — CF + 콜드폴백 + 스티어링 hybrid.py

CF가 항상 base. 콘텐츠는 CF의 warm 결과를 절대 재정렬하지 않음.

  • 콜드폴백(D1): CF가 k개 못 채울 때만 콘텐츠 랭킹으로 보충(품질 게이팅). 설계상 warm recall 불변.
  • 스티어링(F): CF 후보를 신장르(인접 노벨티) / 측면 태그로 재가중.

seed-CF + 프랜차이즈 제외 seed_node

"다크소울 같은 거" → 시리즈 전체(5종)를 seed로(풍부한 co-play 신호) + 명명한 프랜차이즈를 결과에서 통째로 제외(_series_prefix 단어경계 확장). LLM은 한국어→영어 변환만, 정확 매칭은 결정론적 로컬 코드(토큰 0).

multi-entity — interleave 융합 multi_node

"나+친구": 각자 개인화 top을 라운드로빈으로 교차. min/geomean/balanced/interleave 스윕에서 interleave가 유일하게 두 주체를 공평히 살림(친구 B recall 0.028→0.281).

explore — 방향성 스티어링 steer_node NEW

인접 노벨티(사용자 채택): CF가 "그래도 좋아할" 게임 중에서만 방향을 틂(랜덤 신장르 아님, moat 유지).

  • 장르 노벨티: novelty = 1 − taste코사인(콘텐츠 dissimilarity). score = CF · novelty^β. β: low1/med2/high2.5.
  • 측면 스티어링: LLM이 실제 태그 vocab에서 측면 태그 선택(combat/story-rich/atmospheric…) → score = CF · (1+α·aspect).
  • NL에서 {novelty_strength, aspect_tags} 추출, user-score 품질 게이트로 niche≠good 방어.

anonymous — LLM-direct anonymous_node

라이브러리·seed 없는 순수 vibe는 프론티어 LLM이 직접 추천 → 풀로 매핑. 익명 영역은 LLM이 이긴다는 실험 결과에 따른 정직한 위임.

공통 후처리: 제약 · refine · 품질 · 메모리

제약 필터 constraint_filter

로컬 메타데이터(steam_appdetails.csv)로 하드 제약 검증: 협동(2,103)·멀티·싱글·한국어(2,999)·무료·최대가격(≤$10: 6,157)·출시연도. 메타 없으면 보수적으로 드롭.

품질 게이트 (D2) quality_gate

metacritic(30.5% 커버)만으론 70%가 사각 → user-score Bayesian-shrunk 품질(86.6%)을 dense 신호로 추가. 두 신호가 독립 drop-filter.

critic / refine 사이클 critic_node

결과가 k 미만이고 예산 남으면 제약을 정해진 순서로 완화(quality→출시연도→가격×1.5→무료→싱글→멀티→협동→한국어)하고 재필터하는 실제 사이클. 과제약 완결성 0.98 vs 단일패스 0.42.

메모리 played

본/한/싫어한 게임은 세션에 누적되어 추천에서 제외. "봤어요" 버튼으로 표시.

입력: GetOwnedGames / proxy steam_library.py

실시간 = Steam GetOwnedGames(보유 게임+플레이타임, 공개 프로필). 오프라인 데모 = 크롤 데이터의 실제 유저 라이브러리.

평가 방법론

  • 지표 메타-검증: 지표를 쓰기 전에 floor/ceiling, 섭동, 판별력, 교란(인기도) 테스트. 통과 못 하면 강등/디바이어스.
  • 비순환 > 근사 > 가드레일: 행동 co-play 라벨·신장르 holdout처럼 추천기와 독립인 정답 우선. tag-cosine으로 부풀려지는 Genre-Precision은 가드레일로만.
  • 사전등록 결정규칙: 채택/기각 기준을 실험 전에 고정(예: "Δ>0 & 95% CI가 0 제외"). 결과 보고 골대 안 옮김(D, D3가 사례).
  • bootstrap + paired-bootstrap CI: 모든 수치에 신뢰구간. 쌍 비교는 paired.
  • 블라인드·교차 judge: 독립 모델(Claude/Gemini)을 A/B 블라인드로. 자기-baseline 비교로 familiarity-bias 중립화.
  • 귀무·패배 정직 보고: 안 되는 것(96% 패배, D1 null, D3 드롭)을 그대로 기록.
